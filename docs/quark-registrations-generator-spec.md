# `BehaviorRegistrationGenerator` — Design Specification

**Feature:** Per-assembly `QuarkRegistrations` source generator  
**Status:** Design  
**Depends on:** Actor Engine M2 (behavior model, `AddGrainBehavior`, `IActivationMemory`, `IPersistentActivationMemory`)

---

## 1. Problem

After M2, users must manually wire every behavior into DI:

```csharp
// Every assembly with grains repeats this boilerplate:
services.AddGrainBehavior<ICounterGrain, CounterBehavior>();
services.AddGrainTransportDispatcher(new GrainType("CounterGrain"), CounterGrainProxy_TransportDispatcher.Instance);
services.AddScoped<IActivationMemory<CounterState>>(sp =>
    new ActivationMemoryAccessor<CounterState>(
        sp.GetRequiredService<IActivationShellAccessor>()
          .Shell.GetOrCreateHolder<CounterState>()));
```

This is error-prone (easy to forget a registration, easy to get the `GrainType` string wrong) and is exactly what source generators exist to eliminate.

---

## 2. Goal

The generator emits a single `QuarkRegistrations.g.cs` per assembly. The host project calls one method:

```csharp
// Silo startup — replaces all per-grain manual registrations:
silo.Services.AddMyDomainBehaviors();
```

That method registers all behaviors, transport dispatchers, and activation memory accessors found in the assembly.

---

## 3. New generator: `BehaviorRegistrationGenerator`

### 3.1 Trigger

Incremental syntax provider targeting `ClassDeclarationSyntax` nodes where the semantic model confirms the class:

- Is not `abstract`  
- Is not generic (no unbound type parameters)  
- Has at least one constructor that is `public` or `internal`  
- Implements `IGrainBehavior` (directly or transitively via `Quark.Core.Abstractions.Grains.IGrainBehavior`)

### 3.2 Model extraction (per behavior class)

For each qualifying class, extract a `BehaviorModel`:

| Field | How derived |
|---|---|
| `Namespace` | `INamedTypeSymbol.ContainingNamespace` |
| `BehaviorFqn` | Fully-qualified name with `global::` prefix |
| `GrainInterfaceFqn` | First directly-implemented interface that extends `IGrain` (see §3.4) |
| `GrainTypeName` | `[GrainBehavior("key")]` on the class, else `InterfaceName[1..]` (strip one leading `I`) |
| `ProxyClassName` | Same strip rule on `GrainInterfaceFqn` short name, append `Proxy` |
| `InMemoryStateTypes` | All distinct `T` where `IActivationMemory<T>` appears in any constructor parameter |
| `PersistentStateTypes` | All distinct `T` where `IPersistentActivationMemory<T>` appears in any constructor parameter |

### 3.3 Assembly-level collection

All `BehaviorModel` instances are collected via `Collect()` into a single `ImmutableArray`. A single `RegisterSourceOutput` call emits `QuarkRegistrations.g.cs`.

If the collection is empty, emit nothing.

### 3.4 Finding the grain interface

Walk `INamedTypeSymbol.Interfaces` (direct only, not `AllInterfaces`) looking for an interface whose `AllInterfaces` chain contains `IGrain` (`Quark.Core.Abstractions.Grains.IGrain`). 

- If exactly one is found → use it.  
- If zero → emit diagnostic `QRK0010`: behavior class does not implement any IGrain-derived interface.  
- If more than one → emit diagnostic `QRK0011`: ambiguous grain interface; add `[GrainBehavior("typeName")]` to disambiguate, and use the first match as a best-effort fallback so the build still succeeds.

`IGrainObserver` subtypes are **not** grain interfaces for this purpose — skip them.

## 4. Generated output

### 4.1 File: `QuarkRegistrations.g.cs`

```csharp
// <auto-generated/>
#nullable enable
using global::Microsoft.Extensions.DependencyInjection;

[global::System.CodeDom.Compiler.GeneratedCodeAttribute("Quark.CodeGenerator", "0.1.0")]
public static partial class QuarkRegistrations
{
    public static global::Microsoft.Extensions.DependencyInjection.IServiceCollection
        Add{SafeAssemblyName}Behaviors(
            this global::Microsoft.Extensions.DependencyInjection.IServiceCollection services)
    {
        // ── Behavior + transport dispatcher registrations ──────────────────
        {foreach behavior}
        services.AddGrainBehavior<{GrainInterfaceFqn}, {BehaviorFqn}>();
        services.AddGrainTransportDispatcher(
            new global::Quark.Core.Abstractions.Identity.GrainType("{GrainTypeName}"),
            {ProxyFqn}_TransportDispatcher.Instance);
        {/foreach}

        // ── In-memory state accessors (deduplicated per TState) ─────────────
        {foreach distinct InMemoryStateType T}
        services.AddScoped<global::Quark.Core.Abstractions.Hosting.IActivationMemory<{T}>>(sp =>
            new global::Quark.Persistence.Abstractions.ActivationMemoryAccessor<{T}>(
                sp.GetRequiredService<global::Quark.Runtime.IActivationShellAccessor>()
                  .Shell.GetOrCreateHolder<{T}>()));
        {/foreach}

        // ── Persistent state accessors (deduplicated per TState) ────────────
        {foreach distinct PersistentStateType T}
        services.AddScoped<global::Quark.Persistence.Abstractions.IPersistentActivationMemory<{T}>>(sp =>
            new global::Quark.Persistence.Abstractions.PersistentActivationMemoryAccessor<{T}>(
                sp.GetRequiredService<global::Quark.Runtime.IActivationShellAccessor>()
                  .Shell.GetOrCreateHolder<{T}>(),
                sp.GetRequiredService<global::Quark.Persistence.Abstractions.IStorage<{T}>>(),
                sp.GetRequiredService<global::Quark.Runtime.ICallContext>(),
                global::Quark.Persistence.Abstractions.StorageOptions.DefaultStateName));
        {/foreach}

        return services;
    }
}
```

### 4.2 Safe assembly name

Assembly name → strip dots → PascalCase. Examples:

| Assembly | Method name |
|---|---|
| `MyApp.Domain.Grains` | `AddMyAppDomainGrainsBehaviors` |
| `Demo` | `AddDemoBehaviors` |
| `Quark.Tests.Unit` | `AddQuarkTestsUnitBehaviors` |

Non-identifier characters (hyphens, digits at start, etc.) are stripped or replaced as needed.

### 4.3 ProxyFqn derivation

Given `GrainInterfaceFqn = "global::Demo.ICounterGrain"`:

```
shortName  = "ICounterGrain"
proxyShort = shortName.StartsWith("I") ? shortName[1..] + "Proxy" : shortName + "Proxy"
           = "CounterGrainProxy"
proxyFqn   = namespace + "." + proxyShort
           = "global::Demo.CounterGrainProxy"
```

This mirrors the exact rule in `GrainProxyGenerator`, so the reference is always valid when both generators run on the same assembly.

## 5. Deduplication rules

**Behavior entries**: one `AddGrainBehavior` + `AddGrainTransportDispatcher` pair per class. No deduplication needed (each class appears once).

**State accessors**: keyed by the fully-qualified name of `T`. If `CounterBehavior` and `HighSpeedCounterBehavior` both inject `IActivationMemory<CounterState>`, emit `AddScoped<IActivationMemory<CounterState>>` once.

**Persistent vs in-memory**: a `T` that appears in both `IActivationMemory<T>` and `IPersistentActivationMemory<T>` across different behaviors must emit both registrations. They key on `(kind, T)` pairs.

---

## 6. Diagnostics

| ID | Severity | Message |
|---|---|---|
| `QRK0010` | Error | `{ClassName}` implements `IGrainBehavior` but no `IGrain`-derived interface. Add `[GrainBehavior("typeName")]` or implement the grain interface. |
| `QRK0011` | Warning | `{ClassName}` implements multiple `IGrain`-derived interfaces. The first (`{InterfaceName}`) is used. Add `[GrainBehavior("typeName")]` and a single interface to silence. |

## 8. Constraints and non-goals

- **Persistent state name per field**: the generator always uses `StorageOptions.DefaultStateName`. If a behavior needs a custom state name it must register `IPersistentActivationMemory<T>` manually (or a future `[StateName("…")]` attribute, out of scope).
- **Generic behavior classes**: out of scope — emit `QRK0012` warning and skip.
- **Non-public behaviors**: skip silently (they can't be referenced from the generated public method).
- **Cross-assembly observer behavior**: `IGrainObserver` implementations are not grain behaviors. Skipped by the `IGrain` check.
- **`IHostedService`, `IDisposable`, etc. constructor params**: ignored — only `IActivationMemory<T>` / `IPersistentActivationMemory<T>` are special-cased; everything else is assumed already registered by the user.

---

## 9. Impact on existing code

After this generator ships, users replace their manual DI calls with a single generated method. The manual `AddGrainBehavior` extension stays in `RuntimeServiceCollectionExtensions` for test fixtures and edge cases — it is not removed.

The generator test project currently validates `GrainProxyGenerator` output. After this change it also validates `BehaviorRegistrationGenerator` output. Both generators run independently and their outputs don't interfere.

---

## 10. Resolved design decisions

1. **Namespace of `QuarkRegistrations`**: no `namespace` statement — class lives at global scope. Using the assembly's root namespace adds complexity with no practical benefit; `global::` FQNs inside the method body mean the caller can use the method from any namespace without an extra `using`.

2. **`IStorage<T>` generic registration**: `AddInMemoryGrainStorage()` calls `services.TryAddSingleton(typeof(IStorage<>), typeof(InMemoryStorage<>))` — an open-generic registration. `GetRequiredService<IStorage<T>>()` resolves correctly for any `T`. The generator can safely emit `sp.GetRequiredService<global::Quark.Persistence.Abstractions.IStorage<{T}>>()`.

3. **`AddGrainBehavior` already registers transient**: the generator does not emit a separate `services.AddTransient<TBehavior>()` — `AddGrainBehavior` handles it internally.
